# Explicit public DNS for embedding service to bypass host VPN DNS interference

**Date:** 2026-06-16 19:23
**Files:** docker-compose.yml, scripts/smoke-test-dns.sh, Makefile
**Severity:** medium

## Problem

Сервис `embedding` не запускается в Docker-контейнере, если на хост-системе
(Windows) активен VPN-клиент. Контейнер наследует DNS от хост-системы через
Docker Desktop, и DNS-серверы VPN не могут разрешить внешние домены
(huggingface.co, pypi.org), необходимые для загрузки модели sentence-transformers
и других зависимостей во время сборки и при загрузке модели при старте.

Проявление: контейнер падает при старте с ошибкой загрузки модели
`BAAI/bge-small-en-v1.5` из-за невозможности разрешить huggingface.co.

## Root Cause

Docker-контейнеры по умолчанию наследуют DNS-конфигурацию хост-системы
(/etc/resolv.conf внутри WSL2). Когда VPN активен, DNS-серверы хоста —
это DNS-серверы VPN-провайдера, которые:

1. Могут не поддерживать разрешение произвольных внешних доменов
2. Могут быть настроены на блокировку определённых доменов
3. Могут перенаправлять DNS-запросы через VPN-туннель, который недоступен
   из контейнера

Проблема характерна для Docker Desktop на Windows с активным VPN.

## Solution

Добавлена секция `dns:` для сервиса `embedding` в `docker-compose.yml`:

```yaml
embedding:
  build:
    context: ./embedding
    target: production
  dns:
    - 8.8.8.8
    - 1.1.1.1
```

Явное указание публичных DNS-резолверов (Google DNS и Cloudflare DNS)
гарантирует, что контейнер использует их для разрешения доменов,
игнорируя настройки DNS хост-системы.

Docker Compose применяет секцию `dns:` как `--dns` для каждого контейнера,
переопределяя /etc/resolv.conf внутри контейнера.

Дополнительно создан smoke-тест `scripts/smoke-test-dns.sh`, который
верифицирует DNS resolution через явные публичные DNS:

```bash
docker run --rm --dns 8.8.8.8 --dns 1.1.1.1 vedo-embedding:latest python -c "
import socket
import urllib.request
# проверяет DNS и HTTPS до huggingface.co
"
```

## Prevention

- При добавлении новых сервисов, которым нужен доступ к внешним ресурсам
  (загрузка моделей, пакетов, API), всегда добавлять явную секцию `dns:`
  с публичными резолверами, если ожидается работа с активным VPN.
- Smoke-тест `make smoke-dns` следует запускать при изменении сетевой
  конфигурации Docker Compose.
- Альтернативы для Docker Desktop на Windows: настройка `dns:` в
  `daemon.json` (глобально для всех контейнеров) менее предпочтительна,
  так как влияет на все проекты.
