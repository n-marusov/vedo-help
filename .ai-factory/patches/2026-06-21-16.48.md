# PostgreSQL UUID bind mismatch in query history loading

**Date:** 2026-06-21 16:48
**Files:** backend/src/modules/query/service.rs
**Severity:** high

## Problem

POST `/api/query` returned HTTP 500 when a request included `session_id`:

`Internal error: Failed to load history: error returned from database: operator does not exist: uuid = text`

The RAG search and chunk lookup completed successfully, but loading chat history from the
`messages` table failed before the user message could be persisted.

## Root Cause

`QueryService::load_conversation_history` queried PostgreSQL with:

`WHERE session_id = $1`

but bound the parameter as `session_id.to_string()`. The `messages.session_id` column is a
PostgreSQL `uuid`, so PostgreSQL attempted to evaluate `uuid = text`, which has no implicit
operator. Other conversation repository queries bind `Uuid` values directly, so this query was
inconsistent with the schema and with the rest of the backend data-access layer.

## Solution

Changed conversation history loading to bind `session_id` as `uuid::Uuid` instead of `String`.
The raw history query was extracted into a small `pub(crate)` helper so a focused regression test
can exercise the database query directly without going through Chroma, embedding, or OpenRouter.
Added `[FIX]` tracing around history loading success and failure to make future verification from
Docker logs straightforward.

## Prevention

- For PostgreSQL `uuid` columns, bind `uuid::Uuid` directly in sqlx queries; do not stringify UUIDs
  unless the target column is explicitly `TEXT`.
- During review, compare `.bind(...)` parameter types with actual database column types for every
  changed query.
- Keep focused repository/service-level regression tests for database type mismatches, especially
  on paths that are otherwise hidden behind external service calls.
