# Web crawler URL parsing, SSRF, ownership, and page tracking bugs

**Date:** 2026-07-11 21:34
**Files:** backend/src/modules/web_crawl/crawler.rs, backend/src/modules/web_crawl/handlers.rs, backend/src/modules/web_crawl/service.rs, backend/src/modules/web_crawl/repository.rs, backend/migrations/00000000000018_add_failed_status_to_web_crawl_pages.sql
**Severity:** critical

## Problem

The web crawler module (commit `ac670fa`) had 7 critical bugs that made it non-functional and insecure:

1. `extract_origin` returned `"https://"` instead of `"https://host"` — same-domain check always failed, crawler returned 0 pages.
2. `resolve_url` mangled absolute URLs by prepending the base URL to `https://...` links.
3. `is_path_allowed` passed `/admin` when the Disallow rule was `/admin/` (trailing slash mismatch).
4. No SSRF protection — `entry_url` accepted `http://169.254.169.254/...` (AWS metadata), `http://127.0.0.1`, etc.
5. `create_job` did not verify that the user owned the target `collection_id`.
6. `retry_failed_pages` always returned `BadRequest` — status `"failed"` was not in the DB CHECK constraint, and retry reset all pages (not just failed ones).
7. `web_crawl_pages` table was never populated — `get_job_with_pages` always returned an empty list. Additionally, `pages_indexed` counter incremented even when embedding/Chroma indexing failed.

## Root Cause

- **URL parsing bugs (1, 2, 3):** Hand-rolled string manipulation instead of using a proper URL parser. `split('/')` index was off-by-one (`parts[1]` instead of `parts[2]`). Absolute URL passthrough was missing. Trailing-slash matching ignored the case where the path is shorter than the rule.
- **SSRF (4):** The handler validated only the URL scheme but not the host. No blocklist for private/loopback/link-local IP ranges.
- **Ownership (5):** The service created a `CrawlJob` with a user-supplied `collection_id` without checking whether the user owned that collection. The FK constraint validated existence but not ownership.
- **Retry logic (6):** The DB CHECK constraint on `web_crawl_pages.status` did not include `'failed'`, but `retry_failed_pages` filtered for `status == "failed"` — so the count was always 0. Additionally, `update_pages_status_by_job` reset ALL pages, not just failed ones, causing duplicate indexing.
- **Missing page tracking (7):** `start_crawl` saved `Document` + `Chunk` via `doc_repo` but never called `repo.create_page`. The `indexed` counter incremented unconditionally after document save, even if embedding or Chroma indexing failed.

## Solution

1. **`extract_origin`** — Fixed array index from `parts[1]` to `parts[2]`.
2. **`resolve_url`** — Added absolute URL passthrough: `if relative.starts_with("http://") || "https://") { return relative }`.
3. **`is_path_allowed`** — Added trailing-slash matching: if the disallow rule ends with `/` and the path equals the rule without the trailing slash, it's also blocked.
4. **SSRF** — Added `validate_crawl_url()` function that checks scheme + host against `is_private_host()`, which blocks `localhost`, `127/8`, `10/8`, `172.16-31`, `192.168/16`, `169.254/16` (AWS metadata), `0/8`, `::1`, `fe80::`, and IPv4-mapped IPv6. Handler now calls this instead of just `starts_with`.
5. **Ownership** — `create_job` now queries `SELECT user_id FROM collections WHERE id = $1` and returns `Forbidden` if the user doesn't own the collection. Admins bypass the check.
6. **Retry** — New migration `00000000000018` adds `'failed'` to the CHECK constraint. `update_pages_status_by_job` now accepts a `filter_status` parameter to reset only failed pages. `retry_failed_pages` resets job to `'idle'` and calls `start_crawl` to restart the pipeline.
7. **Page tracking** — `start_crawl` now creates a `CrawlPage` record for each crawled page, updates its status to `'indexed'` or `'failed'` based on the outcome of embedding + Chroma indexing. `indexed` counter only increments on successful indexing. Added `update_page_status()` method for individual page updates.

## Prevention

- **Never hand-roll URL parsing** — use a proper URL crate (`url::Url`, `reqwest::Url`) instead of `split('/')` + index arithmetic. Off-by-one errors in string manipulation are extremely common and hard to spot in review.
- **Always implement SSRF validation for user-supplied URLs** that will be fetched server-side. Block private IP ranges, loopback, link-local, and metadata endpoints. A DNS-resolve + IP check is the only airtight approach, as DNS rebinding can bypass hostname-based checks.
- **Verify resource ownership before creating child resources** — FK constraints validate existence, not authorization. Any endpoint that accepts a `collection_id` (or similar foreign key) from user input must check that the requesting user owns the referenced resource.
- **Keep DB constraints in sync with application logic** — if the service filters for `status == "failed"`, the DB CHECK constraint must allow `'failed'`. Integration tests that exercise the full status lifecycle catch this.
- **Track intermediate state in the database** — if an API endpoint promises to show "discovered pages", the service must write page records to the database at crawl time, not just in memory. A separate `pages` table is the source of truth for the API response.
- **Counters must reflect actual success** — incrementing `pages_indexed` when the embedding or Chroma step failed is misleading. Count only what actually succeeded.
- **Filtered updates for retry** — resetting all records (not just failed ones) causes duplicate work. Always scope bulk updates to the relevant subset.
