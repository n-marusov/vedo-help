# Skip pending optimistic message IDs for chat actions

**Date:** 2026-06-21 14:41
**Files:** frontend/src/stores/chat.ts, frontend/src/components/MessageBubble.vue, frontend/src/stores/__tests__/chat.spec.ts
**Severity:** medium

## Problem

The frontend could send edit/delete requests for optimistic chat messages before
those messages were reconciled with persisted backend UUIDs. For example:

`PATCH /api/sessions/77d52d94-d73c-487f-9536-dff47dfa33b8/messages/temp-1782052621000`

The backend route expects `Path<(Uuid, Uuid)>`, so Axum rejected the request
before the handler with:

`Invalid URL: UUID parsing failed: invalid character: found `t` at 0`

## Root Cause

The chat store creates temporary optimistic IDs such as `temp-*` while waiting
for the SSE `done` event to provide persisted `user_message_id` and
`assistant_message_id` values. UI actions and store methods treated every
message ID as eligible for persistence operations, assuming that displayed
messages always had backend UUIDs.

That assumption is false during streaming and reconciliation windows.

## Solution

Added frontend guards for persisted message actions:

- `frontend/src/stores/chat.ts` now detects pending/non-UUID message IDs and
  skips `editMessage` / `deleteMessage` API calls for them.
- `frontend/src/components/MessageBubble.vue` hides edit/delete controls for
  `temp-*` messages and guards action handlers defensively.
- Added `[FIX:chat-temp-id]` warnings so any attempted action on a pending
  optimistic message is visible during debugging.
- Added store tests covering skipped edit and delete behavior for temporary IDs.

## Prevention

- Treat optimistic UI IDs as a distinct state from persisted backend IDs.
- Before calling REST endpoints whose path parameters are UUIDs, validate that
  the client ID is a persisted UUID.
- Hide or disable mutation controls while a message is pending reconciliation.
- Add regression tests for optimistic UI windows, especially around streaming
  and SSE completion.
