# Persist chat session state before slow RAG stages

**Date:** 2026-07-10 00:24
**Files:** backend/src/modules/query/service.rs, frontend/src/stores/__tests__/chat.spec.ts
**Severity:** high

## Problem

Reloading the browser during an in-flight RAG query could leave the newly-created
session visible as `New Chat` with `message_count=0`. SSE recovery itself could
still be available, but the sessions list and direct session fetch showed no user
message because the database had not yet been updated.

## Root Cause

The backend persisted the user message and auto-generated session title only near
the final LLM streaming step. Advanced RAG stages such as embeddings,
Multi-Query, HyDE, search, and reranking can take long enough for the user to
press `F5` before persistence happens. In that sequence the client correctly
restores local recovery state, but the database legitimately still contains a
default-titled session with zero messages.

A second reload during recovery also depends on localStorage state staying intact
when the recovery SSE request is aborted by navigation.

## Solution

Move user-message persistence and default-title replacement to the beginning of
`QueryService::run_pipeline()`, before slow RAG work starts. Load conversation
history before this early persistence so the current query is not duplicated in
the LLM prompt history.

Add `[FIX]` structured logs when the user message is persisted and when the
session is auto-named before the pipeline. Keep assistant-message persistence in
the existing final `done` stream path.

Add regression coverage for title normalization/truncation and for frontend SSE
recovery `AbortError` preserving pipeline localStorage across repeated reloads.

## Prevention

- Persist user-visible conversation state before starting slow external work.
- When moving persistence earlier, audit prompt-history loading to avoid adding
  the current message twice.
- For recovery flows, treat navigation `AbortError` differently from terminal
  failures and preserve localStorage state for the next page instance.
- Review RAG pipeline changes for DB visibility at every user-observable stage:
  sessions list, session details, SSE recovery, and final assistant persistence.
