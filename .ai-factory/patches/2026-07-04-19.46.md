# Missing Local Git Repository Causes Incremental Sync Failure

**Date:** 2026-07-04 19:46
**Files:** backend/src/modules/git_sync/service.rs
**Severity:** medium

## Problem

Incremental git synchronization failed with a "failed to resolve path" error when the local directory for the git repository was missing (e.g., after a Docker container recreation). The system assumed that if the database had a `last_commit_hash` for a repository, the local clone must also exist.

## Root Cause

The `sync_repo` orchestration logic in `backend/src/modules/git_sync/service.rs` checked *only* the database state (`git_repo.last_commit_hash.is_none()`) to determine whether to perform a full clone or an incremental sync. When an incremental sync was chosen, `pull_repo` blindly attempted to open the local repository directory using `git2::Repository::open`. If the directory was missing (because the container's volume didn't persist the git repos, or they were manually cleared), the `open` call failed with a `NotFound` OS error, which was bubbled up as an `InternalError` and caused the entire sync to fail.

## Solution

Modified the `sync_repo` logic to explicitly check for the existence of the `local_path` on the filesystem (`local_path.exists()`). If the database indicates a previous sync (has a commit hash) but the local directory is missing, the service now logs a `[FIX]` warning and gracefully falls back to `full_sync_mode` to re-clone the repository from scratch.

## Prevention

- **Verify filesystem state:** Do not solely rely on database state to infer filesystem state, especially in containerized environments where local caches or ephemeral volumes might be lost across restarts.
- **Graceful degradation:** When a local cache is missing, systems should attempt to recover (e.g., by re-fetching or re-cloning) rather than failing the entire operation.
- **Test missing cache scenarios:** Add integration or unit tests that simulate missing local cache directories for operations that depend on them.
