# Prompt injection hardening and production console guard

**Date:** 2026-06-17 01:52
**Files:** backend/src/shared/llm.rs, frontend/vite.config.ts
**Severity:** medium

## Problem

Two security findings identified during post-implementation security audit:

1. **Prompt injection vulnerability in LLM pipeline** — user input (`prompt`) was concatenated directly into the OpenRouter API messages without any injection-guard delimiters. A user crafting a malicious prompt could override the system instruction and extract arbitrary context or execute unintended LLM behavior.

2. **Production `console.debug` leakage** — 30+ `console.debug` calls in frontend source code were not tree-shaken in production builds, exposing verbose debug logging to production browser consoles.

## Root Cause

1. `build_messages()` in `llm.rs` placed the user prompt directly as the `"user"` role content without wrapping it in injection-guard delimiters or instructing the model to treat it as untrusted input.

2. Vite config had no `esbuild.pure` configuration, so `console.debug` calls survived production builds.

## Solution

1. **Prompt injection guard (`backend/src/shared/llm.rs`):**
   - User prompt is now wrapped in `[USER_QUERY]...[/USER_QUERY]` delimiters
   - The `[/USER_QUERY]` closing tag within the prompt itself is escaped to prevent delimiter injection
   - System instruction expanded with explicit guard instructions telling the model NOT to follow embedded commands in the delimited section
   - Test updated to assert the new wrapping format

2. **Production console guard (`frontend/vite.config.ts`):**
   - Added `esbuild.pure: ['console.debug']` to tree-shake all `console.debug` calls during production builds
   - Dev-mode debugging (`npm run dev`) remains unaffected since esbuild is only used for production bundling

## Prevention

- for LLM-facing code: always wrap user input in injection-guard delimiters AND add explicit system instructions about treating delimited content as untrusted
- for frontend builds: configure `esbuild.pure` as part of project setup, not as an afterthought
- consider a biome lint rule banning `console.debug` without a non-production code path annotation
