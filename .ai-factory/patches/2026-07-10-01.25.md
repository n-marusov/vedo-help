# Pipeline stage lost after repeated reload during recovery

**Date:** 2026-07-10 01:25
**Files:** frontend/src/stores/chat.ts, frontend/src/stores/__tests__/chat.spec.ts
**Severity:** medium

## Problem

After the first `F5`, pipeline recovery restored the stage. After a second or repeated `F5` while the recovery SSE subscription was active, the pipeline stage could disappear on the next load.

## Root Cause

The first reload switches the frontend from the original `/api/query` stream to the recovery `/api/query/:session_id/subscribe` stream. A later reload aborts that recovery stream. Browsers do not always report this as `DOMException` with `AbortError`; it can surface as a generic network `TypeError`. The recovery catch path treated non-`AbortError` failures as terminal recovery failures and cleared the pending pipeline state from `localStorage`, so the next page instance had no active pipeline state to restore.

## Solution

Added a module-level page unload flag set from `pagehide` and `beforeunload` and reset on `pageshow`. `checkPendingPipeline()` now treats any recovery failure during page unload as navigation/reload and preserves `localStorage` instead of clearing it.

Regression coverage now includes:

- a second reload where recovery `fetch` rejects with `TypeError`
- multiple repeated reload disconnects where each new page instance restores the stage and then unloads again

## Prevention

- Do not rely only on `AbortError` to detect reload/navigation aborts.
- For browser reload-sensitive recovery flows, combine abort error checks with lifecycle events such as `pagehide`.
- Tests for reload recovery should include repeated reloads and generic network abort errors, not only explicit `DOMException AbortError`.
