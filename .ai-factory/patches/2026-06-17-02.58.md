# DNS resolution failure for backend container in dev mode (crates.io)

**Date:** 2026-06-17 02:58
**Files:** docker-compose.override.yml
**Severity:** high

## Problem

Backend container in development mode failed to start with `cargo metadata` exiting with:
```
error: failed to get `axum` as a dependency
Caused by:
  [6] Could not resolve hostname (Could not resolve host: index.crates.io)
```

The container couldn't resolve `index.crates.io` (nor any other public host) when `cargo watch -x run` triggered a dependency update.

## Root Cause

The backend service had no explicit `dns:` configuration in `docker-compose.override.yml`. On hosts with a VPN client, the container inherits the VPN's DNS resolver, which blocks resolution of public package registries (crates.io, GitHub, etc.).

The `embedding` service already had explicit DNS (8.8.8.8, 1.1.1.1) with a comment explaining this exact issue, but the same fix was never applied to the `backend` service.

The fundamental assumption was wrong: "Docker containers use host DNS → host has working DNS → containers resolve hosts." This fails when the VPN routes DNS through its own resolver.

## Solution

Added explicit DNS servers to the `backend` service in `docker-compose.override.yml`, matching the pattern already used by the `embedding` service:

```yaml
services:
  backend:
    dns:
      - 8.8.8.8
      - 1.1.1.1
```

This bypasses the host VPN's DNS resolver entirely by using well-known public DNS servers.

## Prevention

- Any Docker Compose service that needs to fetch packages from the internet during development (cargo, pip, npm, apt) must have explicit `dns:` entries.
- When adding a new dev service that downloads dependencies, check if existing services (like `embedding`) have DNS configuration and follow the same pattern.
- Consider adding a global `dns` config at the Compose file level if all services share the same DNS constraint.
