# Fix verification issues for production-ready plan

**Date:** 2026-07-11 15:00
**Files:** scripts/smoke-test.sh, .github/workflows/deploy.yml, grafana/provisioning/dashboards/definitions/cadvisor-container-monitoring.json, backend/tests/health_unit.rs, .ai-factory/DESCRIPTION.md
**Severity:** low

## Problem

Five issues were found during `/aif-verify` of the production-ready implementation:

1. Smoke test (`scripts/smoke-test.sh`) — no per-phase or total elapsed time displayed
2. CI/CD deploy (`.github/workflows/deploy.yml`) — QEMU multi-arch setup missing; `platforms` field absent from build steps, so release job verifies manifests but only `linux/amd64` was built
3. Grafana — no dashboard JSON files shipped in `grafana/provisioning/dashboards/definitions/`, only provisioning skeleton existed
4. Test compilation regression (`backend/tests/health_unit.rs`) — `HealthService::new()` signature changed to accept `Option<NotificationService>`, but test called it with 0 arguments
5. DESCRIPTION.md drift (`.ai-factory/DESCRIPTION.md`) — mentioned SQLite instead of PostgreSQL, missing monitoring/load-testing stack entries

## Root Cause

1. **Smoke test latency:** The original phase reporting had pass/fail and colored output but no timing mechanism. The `wait_for_service` function tracked time implicitly (2s intervals × attempts) but never surfaced elapsed seconds.
2. **QEMU multi-arch:** The deploy workflow was written with the assumption that single-arch (amd64) was sufficient, but the release job verifies multi-arch manifests. QEMU + `platforms` are needed for actual multi-arch builds.
3. **Grafana dashboards:** The provisioning pipeline was wired up (`dashboards.yml`) but `definitions/` directory and dashboard JSON files were not created. The plan mentioned dashboard ID 14282 but the JSON was never downloaded.
4. **Test regression:** `health_unit.rs` was written before the production-ready changes added `notification_service: Option<NotificationService>` to `HealthService::new()`. The constructor was modified but existing tests weren't updated.
5. **DESCRIPTION.md drift:** The project migrated from SQLite to PostgreSQL before the production-ready milestone, but DESCRIPTION.md was never updated. The monitoring stack (Prometheus, Grafana, cAdvisor) was also not listed.

## Solution

1. **Smoke test latency:** Added `GLOBAL_START` timer at startup and `phase_time()` function that computes elapsed seconds/minutes from start. Added `$(phase_time "Phase name")` to each phase's info line and `$(phase_time 'Total')` to the final summary.
2. **QEMU multi-arch:** Added `docker/setup-qemu-action@v3` step after checkout. Added `platforms: linux/amd64,linux/arm64` to both `docker/build-push-action@v6` steps (backend + frontend).
3. **Grafana dashboards:** Created `grafana/provisioning/dashboards/definitions/` directory. Downloaded cAdvisor dashboard (ID 14282) from grafana.com API, replaced `${DS_PROMETHEUS}` template variable with `Prometheus` datasource name for provisioning compatibility.
4. **Test fix:** Changed all 11 occurrences of `HealthService::new()` to `HealthService::new(None)` in `backend/tests/health_unit.rs`.
5. **DESCRIPTION.md:** Updated database from SQLite to PostgreSQL 16. Added monitoring, load testing, health checks, systemd backup timer, and auto-deploy to tech stack. Updated architecture note from "four-service" to "multi-service".

## Prevention

- When changing a public constructor signature in Rust, run `cargo test --no-run` to find all call sites that need updating before committing
- For Grafana dashboard provisioning, add the dashboard JSON at the same time as the provisioning configuration — don't create the pipeline without the payload
- For CI/CD workflows, verify the build step can actually produce what downstream verification steps check (QEMU + `platforms` for multi-arch manifest verification)
- Add a pre-commit check that DESCRIPTION.md tech stack matches actual dependencies
- Include `WARN`-level logging in test binaries when a test is about to fail due to API drift
