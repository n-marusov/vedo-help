# Panic in query endpoint: tokio::sync::Mutex::blocking_lock from async context

**Date:** 2026-06-23 09:40
**Files:** backend/src/modules/query/service.rs
**Severity:** high

## Problem

`POST /api/query` hung and then panicked with:

```
thread 'tokio-rt-worker' panicked at src/modules/query/service.rs:233:51:
Cannot block the current thread from within a runtime. This happens because
a function attempted to block the current thread while the thread is being
used to drive asynchronous tasks.
```

The query was processed correctly through embedding, Chroma search, and
PostgreSQL lookup, but the LLM response stream never completed — the client
request hung until the panic killed the worker.

## Root Cause

`tokio::sync::Mutex::blocking_lock()` was called from inside a synchronous
`.map()` closure on a tokio `Stream`, which ran within the tokio async
runtime. Tokio's Mutex explicitly panics when `blocking_lock()` is called
from within a runtime context because it would block the scheduler thread.

The Mutex was used to accumulate the full LLM response text across stream
chunks. Two sites accessed it:

1. **Line 233** — synchronous `.map()` closure on the stream: called
   `.blocking_lock()` because `.await` is unavailable in sync closures.
2. **Line 248** — async `stream::once` block: called `.lock().await`
   (correct tokio pattern).

Since the critical sections are very short (pushing a string, cloning a
string), a `std::sync::Mutex` is the appropriate type here. No `.await`
points exist while holding the lock, so there is no risk of holding the
lock across a yield point.

## Solution

Changed `tokio::sync::Mutex` to `std::sync::Mutex` in `service.rs`:

- **Import:** `use tokio::sync::Mutex` → `use std::sync::Mutex`
- **Line 233:** `content_tracker.blocking_lock()` → `content_tracker.lock().unwrap()`
- **Line 248:** `fc.lock().await.clone()` → `fc.lock().unwrap().clone()`

`std::sync::Mutex::lock()` blocks the thread (same as `blocking_lock()` in
tokio's API), which is safe here because:
- The lock is held for microseconds (push to String, clone String)
- No `.await` point is crossed while holding the lock

## Prevention

- Do not use `tokio::sync::Mutex` in synchronous stream closures. Use
  `std::sync::Mutex` when the critical section is short and no `.await`
  point is crossed while the lock is held.
- When a Mutex is needed inside both sync and async code, prefer
  `std::sync::Mutex` if lock duration is minimal; otherwise restructure to
  avoid needing the lock in the sync closure (e.g., use a channel sender in
  the map closure and receive in an async task).
- Review all `.blocking_lock()` calls on tokio Mutexes for use-from-async
  violations.
