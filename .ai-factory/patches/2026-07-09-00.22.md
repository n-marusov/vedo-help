# Pipeline state restoration is cleared during page reload abort

**Date:** 2026-07-09 00:22
**Files:** frontend/src/stores/chat.ts, frontend/src/views/ChatView.vue, frontend/src/stores/__tests__/chat.spec.ts
**Severity:** medium

## Problem

When the user refreshed the page (F5) during bot response generation, the frontend still showed the backend default session title "New Chat", lost the pipeline stage indicator, and could not recover the final answer by polling.

The first restoration fix added localStorage-backed pipeline state and `checkPendingPipeline()`, but real browser logs showed that after F5 the reload path had:

- `checkPendingPipeline: called`
- `LS_PIPELINE_ACTIVE=null`
- `checkPendingPipeline: exiting — no pipeline state found`

At the same time, backend logs proved that the query pipeline continued running for the newly created session.

## Root Cause

`sendMessage()` saved pipeline state to localStorage, but then cleared it from a generic cleanup path:

- `AbortError` catch branch called `clearPipelineState()`
- `finally` also called `clearPipelineState()` unconditionally

A browser page reload aborts the active `fetch()`/SSE request. That abort is not a user cancellation; it is exactly the scenario where localStorage state must survive for the next page instance. Because the abort path and `finally` treated reload aborts like completed/cancelled requests, the new page loaded with no recoverable pipeline state.

## Solution

- Added a `streamCancelledByUser` flag to distinguish explicit Cancel button actions from browser navigation/reload aborts.
- `cancelStream()` sets `streamCancelledByUser = true` and still clears localStorage immediately.
- `sendMessage()` now preserves localStorage when it catches `AbortError` that was not caused by `cancelStream()`.
- Removed unconditional `clearPipelineState()` from `finally`; state is cleared only on real terminal conditions such as `done`, stream `error`, explicit user cancellation, recovery completion, 404, or timeout.
- Kept visible `[FIX]` diagnostics around the abort distinction so future reports can show whether state was preserved or cleared.
- Added regression coverage for both cases:
  - reload/navigation-style `AbortError` preserves `chat_pipeline_active`
  - explicit `cancelStream()` clears pipeline state

## Prevention

- Do not put persistent recovery-state cleanup in generic `finally` blocks for long-running streaming requests.
- Treat `AbortError` as ambiguous: it can mean user cancellation, route change, page reload, browser lifecycle shutdown, or network teardown. Track explicit user intent separately.
- Regression tests for reload recovery should assert localStorage contents after an abort, not only restoration after pre-seeded localStorage.
- For streaming UI state, clear recovery state only at known terminal protocol events (`done`, explicit server `error`) or explicit user actions.
