# Pipeline state restoration fails silently when fetchSessions is unavailable

**Date:** 2026-07-09 00:22
**Files:** frontend/src/stores/chat.ts, frontend/src/views/ChatView.vue
**Severity:** medium

## Problem

When the user refreshes the page (F5) during bot response generation, the pipeline state
(session name, pipeline stage indicator, SSE connection) is lost. The original fix added
`checkPendingPipeline()` to restore state from localStorage, but it relied on
`sessions.value` having been populated by a successful `fetchSessions()` API call
in the `onMounted` chain.

If `fetchSessions()` failed or was delayed (token refresh race, network error, slow API),
`sessions.value` would be empty, `sessIdx` would be -1, and the session title restoration
was silently skipped. The toolbar would show "New Chat" instead of the user's query text.

Additionally, all diagnostic logging used `console.debug` which is invisible by default,
making it impossible to verify the fix was even running.

## Root Cause

`checkPendingPipeline()` assumed `sessions.value` was pre-populated by the
`ChatView.onMounted â†’ fetchSessions().finally()` chain. When sessions were empty,
the code skipped title restoration without any fallback, leaving the UI showing the
backend's default "New Chat" title.

The silent `console.debug` logging meant there was no way to confirm:
1. Whether `onMounted` was calling the chain
2. Whether `checkPendingPipeline` was actually running
3. Whether `getPipelineState()` found saved state

## Solution

1. **Synthetic session entry fallback:** When `checkPendingPipeline` cannot find the
   session in `sessions.value`, it now inserts a synthetic `SessionSummary` entry with
   the temp title from localStorage. This ensures the toolbar ALWAYS shows the restored
   title, even if `fetchSessions` hasn't completed or has failed.

2. **Diagnostic logging:** All pipeline-related logging now uses `console.warn` with
   a `[FIX]` prefix, making it visible in the browser console without debug mode:
   - `[FIX] ChatView.onMounted:` â€” chain execution
   - `[FIX] savePipelineState:` â€” confirm state is saved
   - `[FIX] getPipelineState:` â€” confirm state is found on reload
   - `[FIX] checkPendingPipeline:` â€” confirm restoration steps

3. **Error resilience:** Added try/catch around the dynamic `import('@/stores/collections')`
   to prevent a failed import from crashing the entire `checkPendingPipeline` flow (which
   would leave `messages.value` and `activeSessionId` in an inconsistent state).

## Prevention

- Pipeline recovery should be self-sufficient â€” never rely on external API calls having
  completed before restoring visible UI state
- Use `console.warn` (not `console.debug`) for diagnostic logging in recovery code paths;
  `console.debug` is stripped or hidden in many browser configurations
- When inserting synthetic data into reactive arrays, ensure the shape matches the
  TypeScript type to avoid downstream rendering errors
- Test pipeline recovery with mocked `fetchSessions` returning empty sessions array to
  catch the fallback path
