# Fix пre-existing test failures: skeleton timings, export direct API, integration serialization

**Date:** 2026-06-21 15:57
**Files:**
- `frontend/src/views/ChatView.vue`
- `frontend/src/components/DocumentList.vue`
- `frontend/e2e/chat-export.spec.ts`
- `frontend/e2e/loading-skeletons.spec.ts`
- `backend/tests/integration.rs`
**Severity:** medium

## Problem

6 pre-existing test failures across E2E and integration tests:

1. **loading-skeletons: slow session detail** — `v-if="!chatStore.activeSessionId"` в ChatView.vue показывал welcome screen раньше, чем `v-else-if="chatStore.isSessionLoading"`, поэтому skeleton никогда не рендерился
2. **loading-skeletons: slow GET /api/documents** — прямой `pinia.state.value.collections.activeCollectionId = id` не триггерил watcher в Pinia setup store; watcher не имел `{ immediate: true }`
3. **chat-export: Markdown** — `waitForResponse` не видел тело ответа из-за потребления body через `res.blob()` внутри страницы
4. **chat-export: JSON** — та же проблема body consumption
5. **integration: test_conversation_repo_native_uuid_bind** — FK violation: параллельный TRUNCATE через `setup_test_db()` затирал данные параллельного теста
6. **integration: test_git_repo_native_uuid_bind** — та же проблема параллельного TRUNCATE

## Root Cause

1. **Vue template ordering:** `v-if`/`v-else-if` цепочка проверяла `!activeSessionId` первой. Во время `loadSession()` `activeSessionId` ещё null, а `isSessionLoading=true`, но до второго условия цепочка не доходила.
2. **Pinia setup store reactivity:** Прямая мутация `pinia.state.value.collections.activeCollectionId` не триггерит computed-геттер, на который подписан `watch()`. Нужно или вызывать action, или делать `watch(immediate: true)`.
3. **Response body consumption:** `api.exportSession()` вызывает `res.blob()` внутри страницы, что потребляет тело HTTP-ответа. `page.waitForResponse()` в тесте не может прочитать уже потреблённое тело, возвращая пустую строку.
4. **TRUNCATE race condition:** `setup_test_db()` делает `TRUNCATE ... CASCADE` при каждом вызове. Тесты с `#[tokio::test]` выполняются параллельно, и TRUNCATE от одного теста затирает данные другого.

## Solution

1. **ChatView.vue:** Поменял порядок `v-if`/`v-else-if`: сначала `v-if="chatStore.isSessionLoading"`, потом `v-else-if="!chatStore.activeSessionId"`.
2. **DocumentList.vue:** Добавил `{ immediate: true }` к `watch(() => collectionStore.activeCollectionId, ...)`, чтобы fetchDocuments вызывался при монтировании если коллекция уже выбрана.
3. **chat-export.spec.ts:** Заменил `page.waitForResponse()` на прямой вызов `request.fetch()` с токеном, получая ответ независимо от in-page fetch.
4. **loading-skeletons.spec.ts:** Для documents-теста использует `pinia._s.get('collections').setActiveCollection(collectionId)` чтобы триггернуть watcher через store action.
5. **integration.rs:** Добавил `#[serial_test::serial]` на двух тестах, использующих `setup_test_db()`, чтобы предотвратить параллельный TRUNCATE.

## Prevention

- Vue template ordering: скелетон/лоадер должен проверяться раньше, чем empty/fallback-контент.
- Pinia setup stores: `watch` с `{ immediate: true }` для критичных сайд-эффектов.
- E2E: при тестировании API-ответов с body consumption внутри страницы используй `request.fetch()` вместо `waitForResponse`.
- Integration tests с TRUNCATE: маркировать `#[serial_test::serial]` или запускать `--test-threads=1`.
