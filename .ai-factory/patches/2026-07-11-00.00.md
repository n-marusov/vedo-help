# Fix cargo test --lib hang in HTML parsing tests

**Date:** 2026-07-11 00:00
**Files:** backend/src/modules/documents/service.rs, backend/src/shared/notifications.rs
**Severity:** high

## Problem

Running `cargo test --lib` on Windows could hang long enough to freeze the Zed interface when users tried to inspect the last lines with `cargo test --lib 2>&1 | tail -30`. The safe reproduction showed several HTML parsing unit tests running for over 60 seconds. After the hang was removed, the same suite still failed in notification severity-filter tests.

## Root Cause

The HTML parser path called `html2text::from_read(..., 0)`. A wrap width of `0` can make the formatter path non-terminating for small HTML inputs, so simple unit tests such as `test_parse_html_simple` never completed. The shell pipeline was not present in repository scripts; it amplified the user-visible problem by keeping a long-running cargo process connected to Zed's terminal/output handling.

CSV parsing also relied on `reader.records()` to surface UTF-8 errors, but header-only invalid bytes are consumed as headers and can avoid the records iterator. The notification test failures came from comparing `tracing::Level` values directly instead of using an explicit severity ranking.

## Solution

Replaced the HTML parse path with the existing linear `strip_html_tags()` helper and made it available outside tests. Added `[FIX]` debug logs around HTML parsing start/success so future failures expose input size and output length. Added an explicit UTF-8 validation step for CSV before constructing the CSV reader, and corrected the header-only CSV test expectation.

For notifications, added `severity_rank()` and changed the severity filter to compare explicit ranks (`TRACE` < `DEBUG` < `INFO` < `WARN` < `ERROR`) instead of relying on `tracing::Level` ordering.

## Prevention

- Do not pass `0` as a text wrapping width to third-party formatters unless the crate documents that value as valid and terminating.
- Keep unit-test helpers that prove parsing behavior aligned with the production parser path.
- Validate byte encodings explicitly when file formats require UTF-8, instead of assuming iterator consumption will always surface decoding errors.
- Compare logging/severity enums through project-defined semantic ranks when business logic depends on severity ordering.
- On Windows/Zed, prefer bounded command execution with tool-level output limits over shell pipelines such as `2>&1 | tail -30` for potentially long-running test commands.
