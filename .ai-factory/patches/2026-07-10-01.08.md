# Recovery SSE does not restore visible pipeline stage after reload

**Date:** 2026-07-10 01:08
**Files:** backend/src/modules/query/handlers.rs, backend/src/modules/query/service.rs, frontend/src/stores/chat.ts, frontend/src/stores/__tests__/chat.spec.ts
**Severity:** medium

## Problem

After reloading the chat page during an in-flight RAG query, the session title and user message were restored, but the visible pipeline state was not shown or updated. The frontend could recover completion via `/api/query/:session_id/subscribe`, but the UI did not display the current RAG stage during recovery.

## Root Cause

Recovery only waited for terminal job status. The backend `JobStatus::Running` had no stage payload, and the subscribe endpoint emitted only `done` or `error`. On the frontend, `checkPendingPipeline()` also overwrote the saved localStorage stage with a neutral `generating` state and ignored `pipeline_stage` events in the recovery stream.

## Solution

`JobStatus::Running` now carries an optional current stage, and `ActiveJobRegistry::update_stage()` publishes each stage update to active recovery subscribers. The query pipeline calls this method whenever it emits a normal `pipeline_stage` event. The subscribe endpoint now streams the currently known `pipeline_stage`, subsequent stage updates, and finally `done` or `error`.

The frontend now restores the saved stage immediately after reload and updates `pipelineStage` from recovery SSE `pipeline_stage` events. Regression tests cover both synchronous stage restoration and live stage updates during SSE recovery.

## Prevention

Recovery endpoints should preserve user-visible progressive state, not only terminal completion. For long-running SSE workflows, tests should verify both immediate restored UI state from durable browser state and live recovery updates from the backend stream.
