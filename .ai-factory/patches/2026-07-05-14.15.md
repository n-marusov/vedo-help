# Admin session routes missing RBAC middleware and broken SQL parameter numbering

**Date:** 2026-07-05 14:15
**Files:** backend/src/main.rs, backend/src/modules/conversations/handlers.rs, backend/src/modules/conversations/repository.rs
**Severity:** high

## Problem

Two issues in the admin session debug panel:

1. **User list in session filter not populated correctly** — The `/api/admin/sessions/users` and `/api/admin/sessions` endpoints were defined directly on the main Router outside the `admin_router`, so they lacked RBAC middleware. Any authenticated user (not just admins) could access these endpoints. Additionally, the `admin_list_session_users` handler had minimal tracing with no component attribute or result logging.

2. **Broken SQL parameter numbering in `search_sessions`** — The dynamic SQL builder used hardcoded `$1`, `$2`, `$3`, `$4` parameter placeholders, but parameter bindings were conditional. When filters were used selectively (e.g., only `from` and `user_name`), PostgreSQL received SQL like `WHERE 1=1 AND created_at >= $2 AND user_name ILIKE $4` with only 2 bound parameters, causing a database error because `$4` references the 4th parameter position.

## Root Cause

1. **Route registration oversight:** When the admin session endpoints were added during the chat-rework feature, they were placed on the main Router alongside other session routes rather than inside `admin_router` where the RBAC middleware is applied. The developer assumed the general auth middleware was sufficient, overlooking that admin endpoints need the additional role-based gate.

2. **Missing SQL parameter tracking:** The original implementation assumed that all four query parameters (`search`, `from`, `to`, `user_name`) would always be provided together or that the `$N` numbering would magically align with conditional bindings. This assumption breaks whenever any subset of filters is used.

## Solution

### Fix 1: Move admin session routes into `admin_router`

In `main.rs`, added the two admin session routes to the `admin_router` definition:

```rust
let admin_router = Router::new()
    .route("/api/admin/collections", get(...))
    .route("/api/admin/collections/:id", delete(...))
    .route("/api/admin/audit-log", get(...))
    .route("/api/admin/sessions/users", get(conversations_handlers::admin_list_session_users))
    .route("/api/admin/sessions", get(conversations_handlers::admin_list_sessions));
```

Removed the duplicate route definitions from the main Router body. This ensures all admin routes share the same RBAC `require_role("admin")` middleware via the existing `.route_layer()` chain on `admin_router`.

### Fix 2: Fix SQL parameter numbering

Replaced hardcoded `$N` placeholders with a dynamic `param_idx` counter that tracks how many parameters have been bound so far:

```rust
let mut param_idx = 0;
if search.is_some() {
    param_idx += 1;
    sql.push_str(&format!(" AND (title ILIKE ${n} OR ...)", n = param_idx));
}
if from.is_some() {
    param_idx += 1;
    sql.push_str(&format!(" AND created_at >= ${}", param_idx));
}
// ... etc
```

Now the `$N` numbering always matches the actual number of bound parameters, regardless of which filters are used.

### Fix 3: Improved logging

Updated `admin_list_session_users` handler to use structured tracing with `component` attribute and `[FIX]` prefix, plus debug logging of the user count result.

## Prevention

- When adding new admin endpoints, always add them to the `admin_router` definition (not the main Router) so they inherit RBAC middleware automatically.
- When building dynamic SQL with conditional parameters, never use hardcoded `$N` placeholders — use a counter to assign parameter numbers dynamically.
- Add a code review check: every route under `/api/admin/*` must be registered in `admin_router` or have explicit RBAC middleware.
