# Align avatar E2E with UserAvatar task contract

**Date:** 2026-06-16 15:00
**Files:** frontend/e2e/avatar.spec.ts, frontend/playwright.config.ts, frontend/src/router.ts, frontend/src/views/AvatarPreviewView.vue
**Severity:** medium

## Problem

`frontend/e2e/avatar.spec.ts` failed verification for Task 1.2 because it asserted an older avatar contract: initials, online/offline status indicators, and app-level `MessageBubble` selectors. The spec also failed Biome checks and Playwright could not run without a manually started Vite server.

## Root Cause

The E2E test was written against an assumed integrated chat-message avatar behavior, but Task 1.2 only introduced a standalone `UserAvatar` UI component. Integration with `MessageBubble` is planned later in Task 2.1. The Playwright config also assumed a running dev server instead of declaring `webServer`, making focused E2E verification fragile.

## Solution

Added a dedicated `/ui-preview/avatar` route that renders the standalone `UserAvatar` component in user and assistant variants. Rewrote `avatar.spec.ts` to assert Task 1.2 requirements: user person SVG, assistant branded `V` SVG, role-specific token colors, size mapping, and emoji-free rendering. Added Playwright `webServer` configuration so E2E starts Vite automatically. The fix includes `[FIX:avatar-e2e]` debug logging in the preview route and browser console forwarding in the spec.

## Prevention

- Keep E2E assertions aligned with the exact task contract and phase boundaries.
- Use preview routes for standalone UI atoms before they are integrated into larger components.
- Configure Playwright `webServer` whenever tests depend on a local frontend server.
- Run scoped Biome checks on attached QA specs before treating them as verification gates.
