# UTF-8 byte slice panic in generate_title truncation

**Date:** 2026-07-10 16:04
**Files:** `backend/src/modules/conversations/service.rs`
**Severity:** high

## Problem

`generate_title()` panicked with:
```
end byte index 300 is not a char boundary; it is inside 'с' (bytes 299..301 of string)
```

The panic occurred when generating a session title via LLM. The assistant response contained Cyrillic text, and `&response[..300]` sliced at byte index 300 which fell inside a multi-byte UTF-8 character (Cyrillic `с` = 2 bytes at position 299..301).

This caused a 500 error on `POST /api/sessions/:id/generate-title` for any session where the assistant response contained multi-byte characters (Cyrillic, CJK, emoji, accented Latin) and exceeded 300 bytes.

## Root Cause

The code assumed byte-level slicing `&response[..300]` was safe on a UTF-8 `String`. Rust's `Index<Range<usize>>` on `str`/`String` operates on **byte indices**, not character indices. When index 300 falls inside a multi-byte character boundary, Rust panics at runtime. This is a fundamental Rust safety feature — not a bug in Rust, but a bug in the assumption that `len()` returns character count.

This is easy to miss because:
- For pure ASCII text, byte index == character index, so it works fine
- The bug only manifests with multi-byte UTF-8 characters crossing the boundary
- The original developer likely tested with English-only responses

## Solution

Replaced `&response[..300]` with `response.chars().take(300).collect::<String>()`, which operates on Unicode scalar values (characters) and is safe for any UTF-8 text:

```rust
// Before (panic on multi-byte chars):
let truncated = if response.len() > 300 {
    format!("{}...", &response[..300])
} else {
    response.to_string()
};

// After (safe for all UTF-8):
let truncated = if response.chars().count() > 300 {
    tracing::warn!(
        component = "conversations/service",
        response_len = response.chars().count(),
        "[FIX] truncating assistant response for title generation"
    );
    let truncated: String = response.chars().take(300).collect();
    format!("{}...", truncated)
} else {
    response.to_string()
};
```

Also added:
- `tracing::warn!` log with `[FIX]` prefix when truncation occurs
- Unit test `test_generate_title_truncation_with_multibyte_chars` verifying multi-byte safety, boundaries, empty strings, and a loop up to 500 Cyrillic chars to prove no byte-boundary panic

## Prevention

- **Never use `&str[..n]` or `&string[..n]`** for truncation — always use `.chars().take(n)` or `.graphemes()` for grapheme clusters
- When the intent is "first N characters", always check with `.chars().count()` not `.len()`
- Add a Clippy lint or review gate: any `&str` range indexing should be flagged for UTF-8 safety review
- Test string operations with multi-byte characters — one Cyrillic/emoji test case would have caught this immediately
- Consider extracting a helper function like `truncate_chars(s: &str, max: usize) -> String` to prevent similar bugs in other places
