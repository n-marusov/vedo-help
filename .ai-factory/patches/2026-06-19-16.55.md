# Auto-create missing Chroma collection during document indexing

**Date:** 2026-06-19 16:55
**Files:** backend/src/shared/chroma_client.rs
**Severity:** high

## Problem

Document uploads showed progress in the Documents tab, but the uploaded document did not appear in the document list after completion or page reload. Backend logs showed that the document and chunks were saved to SQLite, but Chroma indexing failed:

```text
Chroma add_embeddings failed (HTTP 400 Bad Request): {"error":"InvalidCollection","message":"Collection ce1135b9-b7b5-40c6-a319-a16648089c65 does not exist."}
Upload indexing failed; deactivating document and chunks
```

The service intentionally deactivated the document after indexing failure, so `GET /api/documents?collection_id=...` returned only older active documents.

## Root Cause

The application uses SQLite as the source of collection metadata and Chroma as the vector index. Existing SQLite collections can outlive their matching Chroma collections when Chroma data is reset, recreated, or otherwise loses a collection. The first attempted recovery only called `create_collection(name)`, but Chroma v1 returns `409 Conflict` when the collection name already exists while `/api/v1/collections/{name}/add` still returns `InvalidCollection`. In this Chroma API shape, `add` must be retried against the internal collection `id` returned by `POST /api/v1/collections` with `get_or_create=true`, not necessarily the display `name` used to create it.

## Solution

Updated `ChromaClient::add_embeddings()` to detect Chroma `InvalidCollection` / `does not exist` errors. On the first such failure, it logs a `[FIX]` message, calls `get_or_create_collection_id()` with `get_or_create=true`, extracts the returned internal Chroma `id`, and retries `add_embeddings()` against `/api/v1/collections/{id}/add` within the existing retry loop.

Added regression tests for:

- Missing-collection error detection.
- Extracting the internal Chroma collection `id` from create/get-or-create responses.
- The full recovery sequence: `add` by UUID name fails with `InvalidCollection`, get-or-create returns an internal `id`, and retrying `add` by internal `id` succeeds.

## Prevention

- Treat SQLite/Chroma collection drift as a recoverable condition in indexing paths.
- During code review, check indexing code for assumptions that Chroma state always mirrors SQLite metadata.
- When using Chroma v1 `/api/v1/collections/{collection}/add`, verify whether the endpoint expects an internal collection `id` rather than the configured collection `name`.
- Add integration coverage where Chroma collection metadata exists in SQLite but the first Chroma `add` by name returns `InvalidCollection` and get-or-create returns an internal id.
- Keep using collection UUIDs as Chroma collection names; do not fall back to user-facing collection display names.
