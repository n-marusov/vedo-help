# keycloak-init `apk add gettext` fails on Alpine 3.19 EOL

**Date:** 2026-06-24 18:30
**Files:** docker-compose.yml, docker-compose.test.yml
**Severity:** high

## Problem

The `keycloak-init` service crashed at startup with:
```
WARNING: fetching https://dl-cdn.alpinelinux.org/alpine/v3.19/main: temporary error (try again later)
ERROR: unable to select packages:
  gettext (no such package):
    required by: world[gettext]
```

This blocked the entire `docker compose up` because Keycloak's `depends_on` requires `keycloak-init` to complete successfully before it starts.

## Root Cause

Alpine 3.19 reached end-of-life, and its package repositories have been removed or deprecated. The `apk add gettext` command that installed `envsubst` for template variable substitution failed because the package is no longer served from the Alpine 3.19 repos. Despite already having explicit DNS entries (`8.8.8.8`, `1.1.1.1`) to mitigate VPN DNS issues, the package was simply not available at the upstream repo.

## Solution

Replaced the `envsubst`-based template substitution with `sed`-based inline substitutions, matching the approach already used in `docker-compose.test.yml`. This eliminates the `gettext` package dependency entirely:

- Removed `apk add --no-cache gettext` and `envsubst` calls
- Added `cp /template /import/realm-import.json` followed by 4 `sed -i` calls — one per env var — using the same `|` delimiter pattern as the test compose file
- Updated comments to reflect that DNS is no longer required for template substitution but retained for forward compatibility

## Prevention

- When Alpine base images reach EOL, either upgrade to a newer Alpine version or eliminate the dependency on `apk add` for critical startup logic
- Reuse proven patterns from sibling compose files — `docker-compose.test.yml` already had the `sed`-based approach working reliably
- For template substitution in Docker Compose init containers, prefer `sed` over `envsubst` to avoid runtime package dependencies on EOL base images
