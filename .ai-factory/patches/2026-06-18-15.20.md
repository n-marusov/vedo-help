# KeyCloak port not exposed outside Docker network blocks JWT validation

**Date:** 2026-06-18 15:20
**Files:** docker-compose.yml, docker-compose.override.yml, docker-compose.production.yml
**Severity:** critical

## Problem

All authenticated API endpoints returned 401 with `{"error":{"message":"Invalid or missing token","type":"unauthorized"}}` even after fixing the JWT audience validation. Backend logs showed:

```
WARN JWKS fetch failed (cached keys may be stale):
  JWKS endpoint unreachable: error sending request for url
  (http://localhost:8080/realms/vedo-hub/protocol/openid-connect/certs)
```

The backend (running via `cargo run` on the host) could not reach KeyCloak's JWKS endpoint on `localhost:8080` — connection refused.

## Root Cause

The KeyCloak port `8080` was only mapped in `docker-compose.override.yml`, NOT in the base `docker-compose.yml`. Docker Compose auto-merges the override only when running `docker compose up` without explicit `-f` flags. When the user ran specific services or used a different compose invocation, the port mapping was silently dropped, making KeyCloak inaccessible from the host.

Additionally:
- The production overlay (`docker-compose.production.yml`) did not explicitly close the KeyCloak port, meaning it would leak in production if the override file happened to be merged.
- The override file had the port mapping duplicated from an implicit assumption it would always be merged.

## Solution

**Moved the port mapping to the base `docker-compose.yml`:**

```yaml
# docker-compose.yml
keycloak:
  ports:
    - "${KEYCLOAK_PORT:-8080}:8080"
```

This ensures KeyCloak is always accessible from the host regardless of how Docker Compose is invoked.

**Cleaned up `docker-compose.override.yml`:** Replaced the 8080 mapping with an optional debug port (5005) for KeyCloak development debugging.

**Hardened `docker-compose.production.yml`:** Added explicit `ports: []` for the keycloak service, ensuring the port is NOT exposed in production deployments. This follows the same pattern already used by all other internal services in the production overlay.

## Prevention

- Infrastructure service ports (KeyCloak, Chroma, databases) that need host access MUST be declared in the base `docker-compose.yml`, not in overrides, unless the port is purely a dev-only concern.
- When a port is required for the host-to-container path (e.g., backend running via `cargo run`), it belongs in the base config.
- Production overlays should explicitly close (`ports: []`) any port that should not be publicly accessible.
- Check backend connectivity to KeyCloak early in the startup flow: a failed JWKS fetch is a hard blocker for all authentication.
