# User menu not clickable on first page load due to non-reactive accessToken

**Date:** 2026-06-26 16:48
**Files:** frontend/src/api/client.ts
**Severity:** medium

## Problem

On first page load, the user menu button in AppHeader was not clickable.
After navigating to the admin page and back to chat, it became clickable.
The user saw a non-interactive `<span>` fallback instead of the `<button>`.

## Root Cause

`getAccessToken()` in `api/client.ts` returned a plain `let` module variable.
Vue's reactivity system (computed, watch, ref-based templates) cannot track
changes to plain JavaScript variables — only to `ref()`, `reactive()`, or
Pinia state.

In `AppHeader.vue`, the `userInfo` computed property depends on
`getAccessToken()` to decide whether to render the clickable user menu
button or the static span fallback:

```js
const userInfo = computed(() => {
  const token = getAccessToken();
  if (!token) return null; // ← renders static span
  // ...
});
```

Because `getAccessToken()` was not reactive, if the computed evaluated
before or during `restoreSession()` setting the token (or during async
token refresh), it got `null` — and Vue never re-evaluated the computed
when the token was actually set later.

Navigating to admin triggered store fetches that caused reactive state
(`userName`, `userProvider`) to update, which re-evaluated the computed
— by which point `getAccessToken()` correctly returned the token.

## Solution

Changed `accessToken` from a plain `let accessToken: string | null = null`
to a Vue `ref<string | null>(null)`. Updated `setAccessToken()` and
`getAccessToken()` to use `.value`. Also updated all internal references
in `request()`, `upload()`, and `exportSession()` from `accessToken`
to `accessToken.value` for correct truthiness checks.

Now Vue's reactivity system properly tracks the token and re-evaluates
any `computed()` or template dependency when the token changes.

## Prevention

- Any module-level state shared between the API client and reactive
  components MUST use Vue `ref()` (or Pinia) so computed/watch/template
  dependencies are properly tracked.
- When a computed property depends on a value from a non-reactive module,
  consider wrapping it in a `ref` or reading from Pinia instead.
- Adding a console.debug log in the computed can help diagnose similar
  issues: log whenever `userInfo` re-evaluates.
