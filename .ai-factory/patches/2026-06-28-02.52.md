# Null sources crash in MessageBubble prevents sidebar controls from responding

**Date:** 2026-06-28 02:52
**Files:** frontend/src/components/MessageBubble.vue, frontend/src/components/__tests__/MessageBubble.spec.ts, frontend/src/components/__tests__/ChatWindow.spec.ts, frontend/src/stores/chat.ts, frontend/src/views/ChatView.vue, frontend/e2e/chat-session-switching.spec.ts
**Severity:** high

## Problem

After selecting a session in the sidebar, all sidebar controls (collapse button, new session button, search toggle, session items) stopped responding to clicks. The page appeared interactive but no click handlers fired. A page reload temporarily fixed the issue until another session was selected.

The browser console showed:
```
Uncaught TypeError: Cannot read properties of null (reading 'length')
    at MessageBubble.vue:93
```

This error occurred during Vue's lifecycle hook flush, triggered by `scrollToBottom` -> `nextTick()` -> component mount.

## Root Cause

The backend returned `sources: "null"` (the JSON string `"null"`) for messages without source citations. The `parsedSources` computed property in `MessageBubble.vue` called:

```typescript
return JSON.parse(props.message.sources);
```

`JSON.parse("null")` returns JavaScript `null`, not an empty array. When `onMounted` accessed `parsedSources.value.length`, it crashed on `null.length`, throwing an unhandled TypeError.

In Vue 3, an unhandled error during a lifecycle hook (mounted) causes the error handler to fire via `callWithAsyncErrorHandling`. While Vue catches the error, the corrupted reactive state of the computed property causes subsequent accesses to re-throw, propagating through the component tree and breaking reactivity for sibling controls.

## Solution

Three layers of defense were added:

1. **Array.isArray guard in `parsedSources` computed** (`MessageBubble.vue`): After `JSON.parse()`, check if the result is an actual array. If it's `null` or any non-array (object, number, string), return `[]`. This is the root fix.

2. **try-catch wrapper in `onMounted`** (`MessageBubble.vue`): Wrapped the mounted hook body in try-catch to prevent ANY error from escaping, even from unexpected sources.

3. **Stale request guard in `loadSession`** (`stores/chat.ts`): Added `loadSessionRequestId` counter to ignore stale API responses, preventing race conditions when sessions are switched rapidly.

## Prevention

- Always validate the shape of API responses before accessing nested properties like `.length` on them
- Use `Array.isArray()` before treating a `JSON.parse()` result as an array
- Wrap lifecycle hooks in try-catch when accessing potentially-malformed external data
- Add unit tests for edge cases: messages with `sources: 'null'`, empty sources, undefined sources
- Add integration tests that verify sidebar controls remain functional after session switching, including checking for unhandled JavaScript errors via `page.on('pageerror')`
