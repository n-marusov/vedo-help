# Prevent duplicate chat turns after editing a query

**Date:** 2026-07-10 12:13
**Files:** `frontend/src/stores/chat.ts`, `frontend/src/stores/__tests__/chat.spec.ts`, `backend/src/modules/query/service.rs`
**Severity:** high

## Problem

Editing a user query during or after a RAG response could leave duplicated
conversation turns. The session could contain the edited original user
message, a newly created duplicate user message with the same content, and
multiple assistant answers. The replacement response could also receive the
previous query/answer in prompt history.

## Root Cause

The edit-resend flow treated an edited persisted user message like a brand-new
query. It patched the existing user message, removed the old turn including the
edited user message, and then called the generic send path, which always adds a
new optimistic user message and persists a new backend user message.

There was also a race condition: aborting the frontend stream does not stop the
backend background pipeline. If the old pipeline finished after the edited
replacement pipeline was started, it could still persist a stale assistant
message and reintroduce old history.

## Solution

The frontend edit-resend path now keeps the edited user message as the single
source turn, deletes only following stale messages, and calls `sendMessage` with
an `existingUserMessageId` option so no duplicate user message is appended in
the UI. Backend deletion failures now stop the resend instead of silently
continuing with stale history.

The backend active job registry now assigns a job ID per session pipeline and
only lets the current job publish recovery state or persist the final assistant
message. A superseded background pipeline logs `[FIX] query.stale_assistant_persist_skipped`
and returns `done` without writing a stale assistant response.

## Prevention

- Edit/regenerate flows should preserve or explicitly replace a turn, but not
  mix both strategies.
- Never ignore message deletion failures before re-running a prompt, because
  backend prompt history is built from live session messages.
- Long-running background streams need a server-side supersession guard; client
  abort alone is not enough.
- Regression tests should assert the number of user messages after edit-resend,
  not only that the latest content is correct.
