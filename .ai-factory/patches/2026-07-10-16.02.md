# Duplicate user message on edit-and-resend in chat

**Date:** 2026-07-10 16:02
**Files:** backend/src/modules/query/models.rs, backend/src/modules/query/service.rs, frontend/src/stores/chat.ts, frontend/src/stores/__tests__/chat.spec.ts
**Severity:** high

## Problem

Editing a user query in chat and pressing "resend" caused a duplicate user
message to appear in the session. After `GET /api/sessions/:id`, the session
returned 2 user messages instead of 1: the original edited message plus a
newly created one. The assistant would answer twice (once for each user turn).

## Root Cause

Two independent issues combined:

1. **Backend always creates a new user message on every `POST /api/query`.**
   The `run_pipeline` method unconditionally calls
   `persist_user_message_and_autoname_session`, which inserts a fresh
   `messages` row. There was no mechanism to tell the backend "this is a
   resend — reuse the existing message."

2. **Frontend did not inform the backend about the resend.**
   `editAndResend` called `sendMessage` with `existingUserMessageId`, but
   `sendMessage` only used this flag to skip the *local* temp-message push
   (line 355). It never forwarded the ID to the backend via the POST body,
   so the backend always treated every request as a brand-new query.

The previous fix (pruning stale turns locally and skipping temp user message)
only masked the symptom in the UI but did not prevent the backend from
persisting the duplicate.

## Solution

Added an `existing_user_message_id` field to `QueryRequest` (models.rs).
When the field is set in `run_pipeline`, the pipeline skips
`persist_user_message_and_autoname_session` entirely and uses the existing
message's UUID for the `done` SSE payload instead of a newly created one.

On the frontend, `sendMessage` now passes `existing_user_message_id` in the
POST body when `options.existingUserMessageId` is set.

## Prevention

### Generalizations

1. **Frontend state ≠ backend state.**
   Any state the frontend "knows" (e.g., "this is a resend, not a new query")
   is invisible to the backend unless explicitly transmitted in the API call.
   Implicit assumptions about backend behavior are a frequent source of data
   duplication.

2. **Masking symptoms does not cure the disease.**
   The previous fix (skipping the temp message on the frontend) removed the
   duplicate from the UI but not from the database. A page reload made the
   duplicate visible again. A bug is properly fixed only when the DB state
   is correct, not just the rendered UI.

3. **Every resource-creating POST must be able to opt out of creation.**
   When a scenario exists where the resource may already exist (resend,
   retry, idempotent call), the API must accept an identifier for the
   existing resource and skip creating a new one. This is a specific case
   of POST idempotency.

4. **Two writing operations on the same entity must be coordinated.**
   In this bug, PATCH (edit) and POST (pipeline send) both write to
   `messages`. PATCH updated the existing message, POST created a new one —
   and nothing suppressed the second write. When two endpoints write to the
   same table in the same scenario, they need an explicit contract about
   who creates and who skips.

5. **Optional handler behavior must be expressible in the request schema.**
   If a handler can operate in two modes ("create new" vs. "reuse existing"),
   that option must be a field in the request body. A developer reading the
   handler should not have to guess when it skips the write.

6. **Responsibility boundary: who owns the identifier?**
   The frontend generates temporary IDs (`temp-*`), the backend generates
   real UUIDs. On resend, the frontend already knows the real UUID of the
   existing message. This must be forwarded to the backend so it does not
   generate a second one. The question "whose ID is canonical?" must be
   documented in the API contract and must not allow double generation.

### Code Review Takeaways

- Before merging a fix that "removes duplicates", verify: is the duplicate
  no longer *created* (write-path check), or merely no longer *displayed*
  (read-path check)?
- When adding a field to a response model (`DonePayload`), consider whether
  the corresponding request model (`QueryRequest`) needs a new field so the
  client can control that behavior.
- Any POST that creates an entity and has a "resend" variant should accept
  an optional `existing_<entity>_id` parameter.
- For every frontend → backend data flow, trace the full round-trip:
  action → API call → DB write → API response → UI update. A gap at any
  step creates a consistency bug.
