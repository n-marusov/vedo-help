# Dev stack failed: port 18001 clash and keycloak-init apk fetch error

**Date:** 2026-06-21 13:05
**Files:** `docker-compose.yml`, `docker-compose.test.yml`, `.env.test`
**Severity:** high

## Problem

Running `docker compose --env-file .env up -d` failed at two services:

1. `Container vedo-embedding-1 Error ... Bind for 0.0.0.0:18001 failed: port is already allocated`
2. `Container vedo-keycloak-init-1 Error service "keycloak-init" didn't complete successfully`

The embedding failure aborted the rest of the bring-up; keycloak-init aborted because of an Alpine package fetch error inside its bootstrap script (`apk add gettext`).

The keycloak-init container logs showed:

```
WARNING: fetching https://dl-cdn.alpinelinux.org/alpine/v3.19/main: temporary error (try again later)
ERROR: unable to select packages:
  gettext (no such package):
    required by: world[gettext]
```

## Root Cause

Two independent root causes:

1. **Port collision between dev and test Compose profiles.** The dev override `docker-compose.override.yml` publishes `${EMBEDDING_PORT:-18001}:8001` (default 18001). The test compose `docker-compose.test.yml` publishes `${TEST_EMBEDDING_PORT:-18001}:8001` and `.env.test` hardcodes `TEST_EMBEDDING_PORT=18001`. Every other test service uses a distinct `18xxx`/`13xxx`/`15xxx` host port (chroma 18000, keycloak 18080, backend 13000, frontend 15173, db 15432, openrouter 18002), but the test embedding port unintentionally matched the dev default. A leftover `vedo-test-embedding-1` container from a previous test session was still bound to host 18001, so the new dev embedding container could not bind.

2. **keycloak-init lacked explicit public DNS.** The embedding, backend (dev override), and test embedding services all set `dns: [8.8.8.8, 1.1.1.1]` to avoid host/VPN DNS issues — a documented project pattern. The `keycloak-init` service was the only Alpine-based service that did *not* set this, so when the host's DNS resolution was unreliable (VPN client, transient failure), the runtime `apk add gettext` call could not resolve `dl-cdn.alpinelinux.org` and the init container failed its bootstrap, blocking `keycloak` (which depends on `keycloak-init: service_completed_successfully`).

## Solution

Three minimal, durable changes (operational cleanup of the orphaned test container was also applied to unblock the immediate boot):

1. **`docker-compose.yml`**: added `dns: [8.8.8.8, 1.1.1.1]` to the `keycloak-init` service, mirroring the existing pattern used by embedding/backend. This makes `apk add gettext` resilient to host/VPN DNS breakage.
2. **`docker-compose.test.yml`**: added the same `dns` block to the test `keycloak-init` service for parity, AND changed the test embedding default host port from `18001` → `18081` (consistent with the existing `18xxx` test-port range, distinct from the dev default of `18001`).
3. **`.env.test`**: updated `TEST_EMBEDDING_PORT=18001` → `TEST_EMBEDDING_PORT=18081` so the explicit setting matches the new default and the dev/test stacks no longer collide on that host port.

Operational step taken during the fix: `docker compose --env-file .env.test -f docker-compose.test.yml down` to remove the orphaned `vedo-test-*` containers (volumes preserved) and free port 18001 before re-running the dev bring-up.

## Prevention

- **Distinct port ranges per Compose profile.** Whenever a `docker-compose.<profile>.yml` is added, give every host-published port a default outside the range used by the default/dev stack. Audit `docker-compose.override.yml` and `docker-compose.test.yml` together: any time a `<PROFILE>_ME_PORT` and the dev default resolve to the same number, the dev stack can be blocked by a leftover container from the other profile. Don't rely on operators remembering to `down` the test stack.
- **Apply `dns: [8.8.8.8, 1.1.1.1]` to every container that makes outbound calls at startup.** The project already established this pattern for the embedding and backend dev services — extend it to one-shot init containers too (`keycloak-init`). Any container that does `apk add`, `apt-get install`, `pip install`, `cargo fetch`, or model downloads at startup is brittle without explicit public DNS when a VPN client is active on the host.
- **Avoid runtime package installation in init containers where possible.** Baking `gettext` into a tiny custom image (or switching to a pure-shell substitution like the test `keycloak-init` already does with `sed`) eliminates the network dependency entirely. If the production realm secrets may contain `|`, `&`, or other shell/regex metacharacters, keep `envsubst` (safer) but always pair it with explicit DNS.
- **Verify with `docker compose config`.** After editing any compose file, run `docker compose --env-file .env config --quiet` (and the equivalent with `--env-file .env.test` for the test compose) to catch YAML/interpolation errors before a real `up -d`.
- **Don't leave test stacks running.** After running e2e/integration tests, bring the test stack down (`docker compose --env-file .env.test -f docker-compose.test.yml down`) so its port bindings don't interfere with subsequent dev runs.
