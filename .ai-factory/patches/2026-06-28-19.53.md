# Stale sync lock after server restart

**Date:** 2026-06-28 19:53
**Files:** `backend/src/modules/git_sync/repository.rs`, `backend/src/main.rs`, `backend/src/modules/git_sync/service.rs`
**Severity:** medium

## Problem

After a server restart during an active git repo sync, the repository's DB status was permanently stuck at `"syncing"`. Subsequent sync requests returned `"Sync already in progress for this repository"` even though no process was actually syncing.

Frontend response:
```json
{
    "repo_id": "33a88c52-28f1-43ee-baf7-566a944cbf98",
    "status": "syncing",
    "files_indexed": 0,
    "chunks_total": 0,
    "last_commit": null,
    "error": "Sync already in progress for this repository"
}
```

## Root Cause

The sync lock uses an atomic DB CAS: `UPDATE git_repositories SET status = 'syncing' WHERE id = $1 AND status != 'syncing'`. When the server process is killed during a sync (e.g. `cargo run` triggers a restart, or a crash occurs), the status remains `'syncing'` in the DB forever — there is no TTL, no heartbeat, and no cleanup mechanism.

The lock is purely optimistic and assumes the process holding it will always complete or fail gracefully. This assumption breaks on hard restarts, crashes, or OOM kills.

## Solution

Added `reset_stale_sync_locks()` to `GitRepoRepository` that resets all repositories with `status = 'syncing'` back to `status = 'idle'`. This method is called in `main.rs` during server startup, **before** the scheduler or HTTP handlers can initiate new syncs.

Key design decisions:
- Runs at startup, not on every sync attempt — avoids runtime cost
- Resets **all** stale locks atomically in a single UPDATE — handles multiple repos cleanly
- Logs at WARN level when locks are found, DEBUG when none exist — makes incident investigation easier
- Executes early in the startup sequence, before any other git-sync code path can race with it

## Prevention

- Any process that acquires DB-level locks with a compare-and-swap pattern must have a startup-time cleanup for stale locks if the lock-holding process can be killed without cleanup.
- Log the count of reset stale locks at WARN level so operators notice repeated occurrences (which would indicate frequent crashes).
- Consider adding a `lock_acquired_at` timestamp column in the future for TTL-based lock expiration.

## Follow-up: Leftover clone directory blocks retry

After the stale-lock fix, the next sync attempt failed with:
```
'data/git-repos/33a88c52-28f1-43ee-baf7-566a944cbf98' exists and is not an empty directory
```

The first failed clone left a partial directory on disk. `git2::RepoBuilder::clone()`
refuses to clone into a non-empty directory.

**Fix:** `clone_repo()` now removes any leftover directory inside the `spawn_blocking`
closure before attempting to clone. This ensures clean retries after any failed clone.
