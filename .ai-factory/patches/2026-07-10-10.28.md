# Session title generation skipped after completed chat responses

**Date:** 2026-07-10 10:28
**Files:** frontend/src/stores/chat.ts, frontend/src/stores/__tests__/chat.spec.ts
**Severity:** medium

## Problem

Compact session title generation added in commit `8722c4bbf2e22dc81c159ab1422f32262f3a2585` could be skipped after a user query and assistant response. The normal streaming path only called `api.generateSessionTitle()` when the currently selected `activeSessionId` still existed in the refreshed `sessions` list. If the user switched sessions while the pipeline was running, or if the refreshed list did not contain the active session at that moment, title generation was silently skipped. The reload recovery path did not call `generateSessionTitle()` at all; it only refreshed sessions and assumed the title was already generated elsewhere.

## Root Cause

Title generation was coupled to transient UI selection state instead of the pipeline completion identity. The completed response belongs to `pipelineSessionId`, but the post-stream logic checked `activeSessionId` after the stream ended. `activeSessionId` can legitimately change while streaming because users can browse other sessions during generation. The recovery flow had a separate completion path and duplicated only part of the normal post-completion behavior, so it missed the new LLM title-generation side effect.

## Solution

Added `refreshGeneratedSessionTitle(sessionId)` in `frontend/src/stores/chat.ts` and call it from both completion paths:

- normal `sendMessage()` completion stores `completedSessionId` from `pipelineSessionId` on the `done` event and generates a title for that session after `fetchSessions()`;
- `checkPendingPipeline()` calls the same helper after recovered messages are loaded and sessions are refreshed;
- the helper updates the matching local session title and emits `[FIX]` telemetry events for start, success, empty-title skip, and failure.

Regression tests now verify title generation after a normal bot response, title generation when the user switches sessions mid-stream, and title generation after SSE recovery following reload.

## Prevention

- For streaming pipelines, derive post-completion side effects from the pipeline's stable session id, not from mutable UI selection state.
- Keep reload/recovery completion behavior in sync with the normal completion path; shared helpers are safer than duplicated assumptions.
- Add regression tests for background completion while the user has navigated away from the active pipeline session.
