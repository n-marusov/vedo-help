# JWT audience validation mismatch causes all API endpoints to return 401

**Date:** 2026-06-18 12:00
**Files:** backend/src/shared/auth.rs, keycloak/realm-import.json, backend/tests/auth_middleware_test.rs, frontend/e2e/auth-regression.spec.ts, backend/Cargo.toml
**Severity:** critical

## Problem

All authenticated API endpoints (collections, documents, query, sessions, git-sync, auth) returned 401 with `{"error":{"message":"Invalid or missing token","type":"unauthorized"}}` after successful authentication.

The collection endpoint was the first to be reported, but the issue affected ALL `/api/*` routes protected by the auth middleware.

## Root Cause

The JWT validator in `backend/src/shared/auth.rs` validated the `aud` (audience) claim against `vedo-backend`:

```rust
validation.set_audience(&[&self.client_id]);  // self.client_id = "vedo-backend"
```

However, the frontend (SPA) obtains tokens using the `vedo-frontend` KeyCloak client (public client, PKCE flow). KeyCloak sets the `aud` claim in the access token to the requesting client (`vedo-frontend`), NOT the backend client (`vedo-backend`).

The audience validation was too strict — the issuer + signature checks already provide sufficient security. The audience check was originally added assuming the token would be validated against the backend's own client ID, but the actual auth flow uses a different client for the frontend.

## Solution

**Primary fix (backend):** Removed audience validation from the JWT validator:

```rust
// Before:
validation.set_audience(&[&self.client_id]);

// After:
validation.validate_aud = false;
```

Also removed the now-unused `client_id` field from `JwtValidator` struct.

**Defense-in-depth (KeyCloak):** Added an audience protocol mapper to the `vedo-frontend` client in the realm import, so future tokens will include `vedo-backend` as an additional audience. This way even if audience validation were re-enabled, it would work.

**Root cause justification:** The issuer validation (`set_issuer`) confirms the token was issued by the expected KeyCloak realm. The signature verification (via JWKS) confirms the token hasn't been tampered with. The expiry check prevents replay of expired tokens. These three checks together provide adequate security — audience validation is additive but not essential in this single-realm architecture.

## Prevention

- When using separate KeyCloak clients for frontend (public, PKCE) and backend (confidential), ensure audience validation is configured consistently:
  - Either disable audience validation on the backend (this fix)
  - Or add an audience protocol mapper to the frontend client that includes the backend client ID (defense-in-depth)
- Add regression tests that verify all protected endpoints return 401 without auth (auth_middleware_test.rs)
- Add e2e tests that verify the Authorization header is sent on API calls (auth-regression.spec.ts)
- The `PROTECTED_ROUTES` constant in auth_middleware_test.rs should be updated when new routes are added
