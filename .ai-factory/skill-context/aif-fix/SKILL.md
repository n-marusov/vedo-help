### Chroma Collection Naming Constraints
**Source**: 2026-06-18-19.20.md
**Rule**: When fixing collection CRUD, document indexing, or query logic that calls `ChromaClient::create_collection()`, `delete_collection()`, `add_embeddings()`, or `query()`, never pass user-supplied display names directly to Chroma. Chroma collection names must satisfy: 3-63 chars, start/end with alphanumeric, contain only `[a-zA-Z0-9_-]`, no consecutive periods, and not a valid IPv4 address. User names can contain Unicode, spaces, and punctuation that Chroma rejects with HTTP 400. Use the collection UUID (`id.to_string()`) or a slug derived from it as the Chroma collection name. Store display names only in SQLite. The `QueryService` already uses `collection_id.to_string()` as the Chroma collection name; keep this pattern consistent. Retry logic in `ChromaClient` does not distinguish unrecoverable HTTP 4xx from retryable 5xx, so avoid naming errors upfront.

### Docker Service Discovery Rule
**Source**: 2026-06-18-19.20.md, 2026-06-18-00.50.md
**Rule**: When fixing backend, embedding, frontend proxy, or Docker Compose connectivity issues, distinguish host-facing URLs from container-internal URLs. Inside Docker Compose, containers must call sibling services by Compose service name and container port, for example `http://chroma:8000`, `http://embedding:8001`, `http://backend:3000`, and `http://keycloak:8080`. Do not use `localhost` for backend-to-service calls from inside a container. `localhost` is acceptable only for browser/public issuer URLs, host-published ports, container self-healthchecks, or code/tests that intentionally run on the host. For Compose-managed services, set internal URLs explicitly in the service `environment` block even if application defaults work for local host execution. During review, check every env var ending in `_URL`, `_URI`, `_HOST`, or proxy target for the correct network scope.

### PostgreSQL UUID ≠ TEXT Binding
**Source**: 2026-06-21-16.15.md, 2026-06-21-16.48.md
**Rule**: Any time a `String` parameter is bound to a PostgreSQL `uuid` column, parse to `uuid::Uuid` first with `Uuid::parse_str()`. PostgreSQL has no implicit `text → uuid` cast in `=` / `IN` comparisons. Existing rules cover SQLite TEXT UUID decoding (where columns are TEXT); PostgreSQL uuid columns require the opposite direction. Where possible, prefer `sqlx::query!()` / `query_as!()` macros — their compile-time type checking catches `uuid = text` mismatches before runtime. For dynamic queries, use `Uuid::parse_str()` explicitly before binding.

### tokio::sync::Mutex blocking_lock from Async Context
**Source**: 2026-06-23-09.40.md
**Rule**: Do not use `tokio::sync::Mutex` in synchronous stream closures (`.map()`, `.then()`). `tokio::sync::Mutex::blocking_lock()` panics when called from within a tokio runtime. Use `std::sync::Mutex` when the critical section is short (push/clone String) and no `.await` point is crossed while holding the lock. Review all `.blocking_lock()` calls for use-from-async violations.

### Validation Ordering: Cheap/Global Checks Before Per-Item
**Source**: 2026-06-21-14.55.md
**Rule**: Always order validation so cheap/global checks (entry count, file count, total size limits) run BEFORE per-item validation (MIME, magic bytes, content checks). A malformed ZIP entry failing per-file validation should not return HTTP 415 before the 10-file limit check can return 413.

### Vue Template Ordering: Loader Before Empty/Fallback
**Source**: 2026-06-21-15.57.md
**Rule**: In Vue template `v-if`/`v-else-if` chains, place the loading/skeleton condition BEFORE the empty/fallback condition. If `v-if="isLoading"` is second, the skeleton never renders because the first condition (e.g., `!activeSessionId`) evaluates to true during the async loading window.

### Module-Level State Must Use Vue `ref()`
**Source**: 2026-06-26-16.48.md
**Rule**: Any module-level state shared between an API client and reactive Vue components MUST use `ref()` (or Pinia). Plain `let`/`const` module variables are invisible to `computed()`, `watch()`, and template dependencies. If a computed depends on a value from a non-reactive module, the computed will never re-evaluate when the value changes.
