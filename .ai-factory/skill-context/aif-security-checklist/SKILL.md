## Rules

### LLM Prompt Injection Delimiter Requirement
**Source**: 2026-06-17-01.52.md (enhanced from base SKILL.md)
**Rule**: For this RAG assistant, prompt-injection audits must cover all of:
- User input is never concatenated directly into system prompts
- Input/output boundaries are clearly separated with explicit untrusted-input delimiters, delimiter-closing tokens inside user input are escaped
- The system instruction explicitly tells the model not to follow commands inside delimited user content
- LLM output is treated as untrusted (never executed as code/commands)
- Tool calls from LLM are validated and sandboxed
- Sensitive data is excluded from LLM context
- Rate limiting on LLM endpoints is configured
- Output is filtered for PII/secrets leakage
- Logging and monitoring for anomalous prompts is active

### Production Console Debug Leakage
**Source**: 2026-06-17-01.52.md (enhanced from base SKILL.md)
**Rule**: Frontend security audits must verify:
- Browser/client logs are disabled in production or routed through a logger that no-ops debug output in production
- `console.log`, `console.debug`, `console.info`, and verbose client telemetry are gated by explicit non-production checks (e.g., `if (!isProduction)`)
- In this Vite project, `esbuild.pure` should include `console.debug` or an equivalent production stripping mechanism
- Production UI shows only client-safe error messages; raw exceptions, stack traces, and upstream errors never reach the user
- Client-safe error payloads use stable codes such as `VALIDATION_FAILED`, `UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `CONFLICT`, or `INTERNAL_ERROR`

### KeyCloak/JWT Validation Topology
**Source**: 2026-06-18-12.00.md (enhanced from base SKILL.md)
**Rule**: Auth audits must cover the following JWT validation properties in this project's KeyCloak topology:
- Use RS256 or ES256 (not HS256 for distributed systems with separate resource servers)
- Short access token expiration (15 min), refresh tokens with longer expiry stored securely
- Validate all claims: issuer (`iss`), audience (`aud`), expiry (`exp`), issued-at (`iat`)
- Account for separate KeyCloak frontend and backend clients — do not assume frontend PKCE access tokens use the backend client as audience
- Verify issuer, signature/JWKS, expiry, audience policy, and optional audience mapper configuration together
- Implement token revocation capability
- Never store sensitive data in JWT payload
- Treat a broken JWKS fetch or universal 401 on protected routes as critical
- Store refresh tokens securely (httpOnly cookie pattern)
